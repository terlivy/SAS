# SAS 版本治理规范

> 本文件定义了 SAS 准则的版本控制流程。所有版本变更必须遵循此规范。

---

## 1. 版本号定义

```
格式：v主版本.次版本.修订号

v1.6.0
│ │ └─ 修订号：小修复、文字调整
│ └─ 次版本：新增章节、流程变更
└─ 主版本：重大架构变更
```

| 修订级别 | 触发条件 | 例子 |
|---------|---------|------|
| **主版本** | 重大架构变更 | SAS v1.5 → v2.0 |
| **次版本** | 新增章节、流程变更 | v1.6.0 → v1.7.0 |
| **修订号** | 修复/微调 | v1.6.0 → v1.7.1 |

---

## 2. 版本文件规范

```
04-工作准则/
├── Snoopy Agent Smart (SAS)工作准则-v1.0.md   # 历史版本（只读）
├── Snoopy Agent Smart (SAS)工作准则-v1.4.md   # 历史版本（只读）
├── Snoopy Agent Smart (SAS)工作准则-v1.5.md   # 当前生效版本
└── CHANGELOG.md                              # 所有版本变更记录
```

**规范：**
- 每个版本一个 md 文件
- 文件名为 `v{版本号}.md`
- 历史版本标记为只读 `[归档]`
- 当前生效版本在文件名中标注 `(当前)`

---

## 3. 版本发布流程（强制六阶段）

### 阶段 1：提案
- 由 SAS-SOP 专家 / 人类 提出版本需求
- 说明：版本号、变更内容、理由

### 阶段 2：评审
- 人类审批：变更内容是否合理
- 评估：对 SC 行为的影响

### 阶段 3：编写
- 更新 `v{x}.md` 文件
- 填写 `CHANGELOG.md`
- 更新 `git tag v{x}.0`

### 阶段 4：验证
- 测试变更对 SC 的影响
- 确保 SOUL.md 引用正确的版本

### 阶段 5：发布
- 更新 SOUL.md 的版本引用（强制！）
- git commit + tag
- 推送 GitHub

### 阶段 6：归档
- 旧版本文件标记为只读
- 发布公告

---

## 4. SOUL.md 版本同步（强制）

**每次版本发布必须同时更新 SOUL.md：**

```markdown
# 当前 SAS 版本：v1.6.0
```

**禁止：** 改了版本文件但不更新 SOUL.md

## 6. git tag 规范

```bash
# 发布版本
git tag -a v1.6.0 -m "Release SAS v1.6.0 - 新增Skill安全审核规则"
git push origin v1.6.0

# 查看所有版本
git tag -l
```

## 7. 版本状态定义

| 状态 | 说明 |
|------|------|
| `提案中` | 正在评审，未发布 |
| `待发布` | 评审通过，待执行发布流程 |
| `已发布` | 完成所有发布阶段，SOUL.md 已更新 |
| `已归档` | 历史版本，只读 |

---

## 8. 版本控制权限

| 动作 | 权限 |
|------|------|
| 创建新版本文件 | SC 或 人类 |
| 修改当前版本 | SC 或 人类（通过发布流程） |
| 修改历史版本 | ❌ 禁止 |
| 更新 SOUL.md | SC（强制）或 人类 |
| 创建 git tag | SC 或 人类 |
| 审批版本发布 | **仅人类** |
