## 🎯 第一章：SAS 核心理念

### 1.1 慢即是快 (Slow is Fast)

**定义**：通过充分的规划、设计和确认，避免执行阶段的反复修改和返工，从而实现整体效率的最大化。

**具体体现**：
1. **设计阶段慢**：花时间设计清晰、可复用的工作流模板
2. **确认阶段慢**：充分沟通，确保需求理解一致
3. **规划阶段慢**：制定详细的实施计划和风险评估
4. **执行阶段快**：基于完善规划的高效执行
5. **复用阶段快**：基于模板和案例库的快速复制

### 1.2 系统化思维 (Systematic Thinking)

**定义**：将每个任务视为系统的一部分，关注任务间的依赖关系、数据流转和整体优化。

**核心原则**：整体大于部分、可复用性、可扩展性、可维护性、可观测性

### 1.3 三层智能体协作架构

#### 第一层：用户层
- **角色**：需求提出者、决策审批者、价值评估者
- **职责**：提出业务需求、审批准则更新、评估工作成果

#### 第二层：主脑层
- **角色**：SAS工作准则执行者、任务协调者、质量保证者
- **职责**：执行准则、协调智能体、确保质量、向用户汇报

#### 第三层：迭代智能体层（SAS-SOP专家）
- **角色**：准则优化者、知识研究者、流程改进者
- **职责**：分析执行效果、智能搜索研究、提出优化方案、版本管理

**协作流程**：
```
用户需求 → 主脑接收 → 参照SAS准则执行 → 结果反馈
      ↑                           ↓
      └── SAS-SOP专家分析 ──→ 提出准则优化 → 用户审批 → 主脑执行新准则
```

### 1.4 智能体协作模式

| 角色 | 职责 |
|------|------|
| 主脑架构师 | 整体规划和架构设计 |
| 专业执行者 | 具体领域的专业执行 |
| 质量保证者 | 测试和验证 |
| 知识管理者 | 文档和知识管理 |
| 财务监控者 | 成本控制和财务记录 |

### 1.5 SAS-SOP专家与主脑分工协议

**背景**：SAS-SOP每日准则优化cron任务（每日16:00 UTC）负责触发优化流程，但实际优化分析工作由主脑执行。本协议明确分工边界。

#### 分工原则

| 维度 | SAS-SOP专家（Cron任务） | 主脑（SC） |
|------|-------------------------|------------|
| **触发** | ✅ 每日按时触发优化流程 | — |
| **分析** | ❌ 仅发出分析指令 | ✅ 执行深度分析 |
| **建议** | ❌ 仅提出优化方向 | ✅ 输出具体可执行方案 |
| **撰写** | ❌ | ✅ 撰写优化文档 |
| **审核** | ❌ | ✅ 审核后提交用户 |
| **执行** | ❌ | ✅ 用户审批后执行新准则 |

#### 执行流程

```
┌─────────────────────────────────────────────────────────┐
│  SAS-SOP专家 Cron（每日16:00 UTC）                        │
│  1. 发出优化触发消息                                     │
│  2. 提供当日执行数据分析（从AI Monitor读取）              │
│  3. 提出初步优化方向（不超过3条）                        │
└─────────────────────┬───────────────────────────────────┘
                      ↓
┌─────────────────────────────────────────────────────────┐
│  主脑（SC）接收任务                                      │
│  1. 分析SAS准则当前版本的不足点                          │
│  2. 结合cron提供的数据和方向                             │
│  3. 输出完整优化方案（建议≥3条）                        │
│  4. 更新准则文档（新建版本）                             │
│  5. 在飞书群汇报优化结果，等待用户审批                    │
└─────────────────────────────────────────────────────────┘
```

#### 每日优化任务要求

- **时间要求**：主脑应在cron触发后**2小时内**完成分析并输出结果
- **输出要求**：版本号递增、变更说明明确、优化内容具体可执行
- **质量要求**：每次优化不少于3条具体建议，每条说明预期收益

#### 2.2.3 评分维度与标准

| 维度 | 权重 | 评分标准（1-10） |
|------|------|-------------------|
| **速度** | 25% | 响应时间、执行效率 |
| **质量** | 25% | 输出正确性、完整性 |
| **问题率** | 25% | 出错次数、返工次数（越低越好） |
| **成本** | 25% | Token消耗、时间成本 |

**综合评分** = 速度分×0.25 + 质量分×0.25 + (10-问题分)×0.25 + 成本分×0.25

#### 2.2.4 四项核心要求

| 要求 | 含义 | 达标标准 |
|------|------|----------|
| **快** | 响应快、执行快、交付快 | 实际时间≤计划时间 |
| **准** | 理解准、执行准、交付准 | 一次通过率≥90% |
| **问题少** | 预判准、风险低、少返工 | 返工次数≤1次/任务 |
| **成本低** | Token省、时间省、资源省 | 实际成本≤预算80% |

### 3.3 智能体身份固化规范 (v1.3.0 新增)

**问题背景**：每次任务生成新UUID，缺乏历史连续性，难以追踪智能体长期表现。

#### 3.3.1 ID架构

| ID类型 | 格式 | 说明 |
|--------|------|------|
| **Agent ID** | 固定字符串 | 如 `sas-main`, `sas-leader`, `xiaowen`, `hr-agent` |
| **Session ID** | UUID | 每次会话生成，用于会话追踪 |
| **Task ID** | UUID | 每个任务生成，用于任务追踪 |

#### 3.3.2 配置要求

- 每个智能体必须在 `agent.json` 中指定固定的 `name` 字段
- Session ID 和 Task ID 仅用于日志和追踪，不作为智能体身份标识
- 主脑应维护"Agent ID → 能力映射表"

### 4.1 任务生命周期管理规范 (v1.3.0 新增)

**目的**：解决任务链断裂（pending状态无人接管）问题

#### 4.1.1 任务状态机

```
pending → running → completed
              ↓
           failed → 可重试?
              ↓
           cancelled
```

| 状态 | 定义 | 超时限制 |
|------|------|----------|
| **pending** | 任务已创建，等待智能体接管 | **≤2小时**，超时告警 |
| **running** | 智能体正在执行 | ≤计划时间×1.5 |
| **completed** | 任务成功完成 | — |
| **failed** | 任务执行失败 | — |
| **cancelled** | 任务被取消 | — |

#### 4.1.2 任务看门狗机制

| 检查项 | 阈值 | 动作 |
|--------|------|------|
| **pending超时告警** | >2小时 | 发送告警到飞书群 |
| **pending自动升级** | >6小时 | 触发自动重试或升级处理 |
| **running超时预警** | >计划×1.2 | 发送预警通知 |
| **running超时中断** | >计划×1.5 | 暂停任务，等待人工决策 |

#### 4.1.3 自动恢复机制

**原则**：会话重置后，未完成任务必须在下次启动时自动恢复

```
任务恢复流程：
1. 检测重置事件（通过 commands.log 或 heartbeat 异常）
2. 查询任务队列状态（/api/harness/tasks）
3. 识别 pending/failed 状态任务
4. 评估恢复可行性（超时时间、错误原因）
5. 执行恢复动作：
   - pending>6h → 重新创建任务
   - pending<6h → 等待原智能体恢复
   - failed → 分析错误原因后重试
6. 记录恢复日志
```

### 4.2 智能体空转检测规范 (v1.3.0 新增)

**目的**：解决智能体空转（大量心跳快照但0 Token工作产出）问题

#### 4.2.1 空转定义

**空转状态**：连续3小时快照数>100但Token输入=0

#### 4.2.2 空转检测指标

| 指标 | 正常范围 | 告警阈值 |
|------|----------|----------|
| 快照频率 | 10-50个/小时 | >100个/小时 |
| Token输入/快照 | >100 Token/快照 | =0 Token/快照 |
| 连续空转时长 | 0 | >3小时 |

#### 4.2.3 空转处置流程

```
检测到空转 →
    ↓
1. 记录空转起始时间和快照数
2. 发送告警到飞书群（包含持续时长和快照数）
3. 检查任务队列状态
4. 检查 API 连通性
5. 如持续>6小时，尝试触发任务重新分配
6. 记录诊断信息和处置动作
```

### 4.3 会话重置恢复SOP (v1.3.0 新增)

**背景**：86%的会话重置由 `gateway:sessions.reset` 自动触发，需标准恢复流程

#### 4.3.1 五步恢复流程

| 步骤 | 操作 | 目标状态 |
|------|------|----------|
| **1. 检测** | 监控 commands.log 中的 `gateway:sessions.reset` 事件 | 识别重置 |
| **2. 快照** | 保存当前任务队列快照到内存/文件 | 防止丢失 |
| **3. 等待** | 等待30秒确认重置完成 | 稳定状态 |
| **4. 恢复** | 从 memory/sessions 加载上下文，继续未完成任务 | 继续执行 |
| **5. 验证** | 确认任务链完整性和Token产出 | 恢复正常 |

#### 4.3.2 重置原因分类

| 原因 | 占比 | 处理策略 |
|------|------|----------|
| gateway:sessions.reset | ~86% | 自动恢复，检查任务链 |
| 飞书渠道重置 | ~5% | 检查消息丢失 |
| 人工重置 | ~5% | 确认任务状态 |
| 其他 | ~4% | 按情况分析 |

### 4.4 任务失败处理流程

```
任务执行失败 → 记录失败信息+错误日志 → 评估失败原因
    ↓
├── 可重试（网络超时、偶发错误）
│       → 等待5分钟后重试（最多3次）
│       → 重试成功 → 继续执行
│       → 重试失败 → 进入不可重试流程
│
├── 需人工介入（逻辑错误、配置问题）🐾主脑负责
│       → 立即暂停任务
│       → 记录详细错误信息
│       → 通知用户说明问题
│
└── 需修改方案（需求不清、技术不可行）🐾主脑负责
        → 暂停任务
        → 提出替代方案
        → 等待用户确认
```

### 4.5 预算超支处理

| 预算消耗 | 等级 | 处理方式 |
|----------|------|----------|
| <80% | 🟢 正常 | 正常执行，注意监控 |
| 80-90% | 🟡 预警 | 发出预警通知，使用低成本模型 |
| 90-95% | 🟠 警戒 | 暂停非关键任务，重要任务需确认 |
| >95% | 🔴 紧急 | 只执行紧急任务，立即通知用户 |
| 超支 | ⚠️ 超出 | 停止所有非授权任务，等待用户指示 |

### 4.6 响应超时处理

| 超时情况 | 处理方式 |
|----------|----------|
| API响应超时2分钟 | 自动重试（最多2次） |
| 重试仍超时 | 标记"卡住"状态，通知用户 |
| 30分钟无法恢复 | 提供替代方案或降级方案 |

### 6.3 Token产出监控规范 (v1.3.0 新增)

