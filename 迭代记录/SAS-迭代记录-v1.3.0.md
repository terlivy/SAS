## 二、问题分析

### 2.1 P0 问题（紧急）

#### 问题 A：任务链断裂

**现象**：5 条任务中 4 条长期处于 pending 状态，最后更新均为 2026-03-27 21:xx

**根因**：
1. 任务提交后没有智能体认领
2. pending 状态无超时监控机制
3. 主脑无法感知任务链卡滞

**影响**：任务无法完成，用户感知为"任务丢失"

#### 问题 B：智能体空转

**现象**：2026-03-28 全日 1,353 个心跳快照，0 Token 工作产出

**根因**：
1. 会话重置后智能体失去上下文
2. 智能体退化为心跳待机模式
3. 无空转检测机制

**影响**：API 成本浪费（1,353 次心跳），无实际产出

### 2.2 P1 问题（重要）

#### 问题 C：会话重置无恢复

**现象**：86% 的会话重置由 `gateway:sessions.reset` 自动触发，但无标准恢复流程

**根因**：
1. 重置事件未被监控
2. 重置后任务队列未保存
3. 上下文恢复依赖手动操作

#### 问题 D：Token 输出统计缺失

**现象**：6 个模型中 5 个输出 Token 为 0

**根因**：可能是 Token 统计链路设计缺陷，需排查 `/api/ledger/by-model` 数据

### 2.3 P2 问题（一般）

#### 问题 E：智能体 ID 不稳定

**现象**：每次任务生成新 UUID，难以追踪智能体历史表现

**根因**：`agent.json` 中 name 字段未作为固定标识使用

## 三、优化建议

### 3.1 🔴 P0 建议（已采纳）

#### 建议 A：任务生命周期管理规范

**内容**：
- 定义任务状态机：pending → running → completed/failed/cancelled
- pending 状态不得超过 2 小时，超时告警
- pending 超过 6 小时触发自动重试或升级

**预期收益**：消除任务链断裂问题，提升任务完成率

#### 建议 B：智能体空转检测规范

**内容**：
- 定义空转阈值：连续 3 小时快照 >100 但 Token 输入 =0
- 检测到空转立即告警
- 持续 6 小时以上触发任务重新分配

**预期收益**：消除空转浪费，提升 Token 产出效率

### 3.2 🟠 P1 建议（已采纳）

#### 建议 C：会话重置恢复 SOP

**内容**：五步标准恢复流程
1. 检测重置事件
2. 保存任务队列快照
3. 等待 30 秒确认
4. 从 memory/sessions 加载上下文
5. 验证任务链完整性

**预期收益**：会话重置后可自动恢复，降低人工干预

#### 建议 D：Token 产出监控规范

**内容**：
- 输出 Token 连续 2 小时 =0 告警
- 每快照最低输入 Token 50
- 缓存命中率低于 30% 告警

**预期收益**：及时发现 Token 统计问题

### 3.3 🟡 P2 建议（已采纳）

#### 建议 E：智能体身份固化规范

**内容**：
- 每个智能体使用固定 name 作为 Agent ID
- Session ID 和 Task ID 仅用于日志追踪
- 主脑维护 Agent ID → 能力映射表

**预期收益**：可追踪智能体历史表现
