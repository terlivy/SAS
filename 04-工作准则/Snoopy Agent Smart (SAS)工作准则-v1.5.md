## 🎯 第一章：SAS 核心理念

### 1.1 慢即是快 (Slow is Fast)

**定义**：通过充分的规划、设计和确认，避免执行阶段的反复修改和返工，从而实现整体效率的最大化。

**具体体现**：
1. **设计阶段慢**：花时间设计清晰、可复用的工作流模板
2. **确认阶段慢**：充分沟通，确保需求理解一致
3. **规划阶段慢**：制定详细的实施计划和风险评估
4. **执行阶段快**：基于完善规划的高效执行
5. **复用阶段快**：基于模板和案例库的快速复制

### 1.2 系统化思维 (Systematic Thinking)

**定义**：将每个任务视为系统的一部分，关注任务间的依赖关系、数据流转和整体优化。

**核心原则**：整体大于部分、可复用性、可扩展性、可维护性、可观测性

### 1.3 三层智能体协作架构

#### 第一层：用户层
- **角色**：需求提出者、决策审批者、价值评估者
- **职责**：提出业务需求、审批准则更新、评估工作成果

#### 第二层：主脑层
- **角色**：SAS工作准则执行者、任务协调者、质量保证者
- **职责**：执行准则、协调智能体、确保质量、向用户汇报

#### 第三层：迭代智能体层（SAS-SOP专家）
- **角色**：准则优化者、知识研究者、流程改进者
- **职责**：分析执行效果、智能搜索研究、提出优化方案、版本管理

**协作流程**：

用户需求 → 主脑接收 → 参照SAS准则执行 → 结果反馈
↑ ↓
└── SAS-SOP专家分析 ──→ 提出准则优化 → 用户审批 → 主脑执行新准则

### 1.4 智能体协作模式

| 角色       | 职责               |
| ---------- | ------------------ |
| 主脑架构师 | 整体规划和架构设计 |
| 专业执行者 | 具体领域的专业执行 |
| 质量保证者 | 测试和验证         |
| 知识管理者 | 文档和知识管理     |
| 财务监控者 | 成本控制和财务记录 |

### 1.5 SAS-SOP专家与主脑分工协议

**背景**：SAS-SOP每日准则优化cron任务（每日16:00 UTC）负责触发优化流程，但实际优化分析工作由主脑执行。本协议明确分工边界。

#### 分工原则

| 维度     | SAS-SOP专家（Cron任务） | 主脑（SC）             |
| -------- | ----------------------- | ---------------------- |
| **触发** | ✅ 每日按时触发优化流程  | —                      |
| **分析** | ❌ 仅发出分析指令        | ✅ 执行深度分析         |
| **建议** | ❌ 仅提出优化方向        | ✅ 输出具体可执行方案   |
| **撰写** | ❌                       | ✅ 撰写优化文档         |
| **审核** | ❌                       | ✅ 审核后提交用户       |
| **执行** | ❌                       | ✅ 用户审批后执行新准则 |

#### 执行流程

┌─────────────────────────────────────────────────────────┐
│ SAS-SOP专家 Cron（每日16:00 UTC） │
│ 1. 发出优化触发消息 │
│ 2. 提供当日执行数据分析（从AI Monitor读取） │
│ 3. 提出初步优化方向（不超过3条） │
└─────────────────────┬───────────────────────────────────┘
↓
┌─────────────────────────────────────────────────────────┐
│ 主脑（SC）接收任务 │
│ 1. 分析SAS准则当前版本的不足点 │
│ 2. 结合cron提供的数据和方向 │
│ 3. 输出完整优化方案（建议≥3条） │
│ 4. 更新准则文档（新建版本） │
│ 5. 在飞书群汇报优化结果，等待用户审批 │
└─────────────────────────────────────────────────────────┘

#### 每日优化任务要求

- **时间要求**：主脑应在cron触发后**2小时内**完成分析并输出结果
- **输出要求**：版本号递增、变更说明明确、优化内容具体可执行
- **质量要求**：每次优化不少于3条具体建议，每条说明预期收益

### 1.6 CEO 模式协作架构（🔄 v1.5.0 新增）

**背景**：用户的核心诉求是"按计划时间拿结果"，而非被执行细节频繁打扰。CEO 模式将用户定位为决策者，执行过程中的问题由 Leader 自主处理。

#### 1.6.1 四层角色定义

| 层级 | 角色 | 类比 | 核心职责 | 用户交互 |
|------|------|------|---------|---------|
| **L0** | 用户（User） | CEO | 提需求、审批计划、审批方案、查看成绩单 | **只在两个决策点参与** |
| **L1** | SC（主脑） | PM（项目经理） | 制定计划、设计方案、指派 Leader、质量终审 | 用户的唯一对话窗口 |
| **L2** | Leader Agent | 项目经理（Team Lead） | 分解子任务、分配 Worker、盯进度、处理执行问题 | **不直接联系用户** |
| **L3** | Worker Agent | 执行者（Developer） | 按分配任务执行，产出代码/文档/配置 | **不直接联系用户** |

#### 1.6.2 用户参与的两个决策点

```
┌─────────────────────────────────────────────────────────┐
│ 决策点 1：计划审批（阶段一门）                           │
│ ─────────────────────────────────────                   │
│ SC 产出 → 用户审批 → 通过/驳回/修改                     │
│ 用户只看：需求理解是否准确、目标是否对齐、时间估算是否合理│
└─────────────────────────────────────────────────────────┘
                        ↓ 通过
┌─────────────────────────────────────────────────────────┐
│ 决策点 2：方案审批（阶段二门）                           │
│ ─────────────────────────────────────                   │
│ SC 产出 → 用户审批 → 通过/驳回/修改                     │
│ 用户只看：技术路线是否合理、Agent 分配是否恰当、风险评估  │
└─────────────────────────────────────────────────────────┘
                        ↓ 通过
┌─────────────────────────────────────────────────────────┐
│ 执行区（用户不参与）                                    │
│ ─────────────────────────────────────                   │
│ Leader 全权负责：分解任务、分配 Worker、进度监控、        │
│ 问题处理、模型调配、子任务回退                           │
│ 唯一例外：遇到超出 L2 权限的问题 → 上报 SC → SC 决定    │
│ 是否需要打扰用户（见 §2.6 Leader 自治规则）              │
└─────────────────────────────────────────────────────────┘
                        ↓ 完成
┌─────────────────────────────────────────────────────────┐
│ 成绩单交付                                               │
│ ─────────────────────────────────────                   │
│ Leader 产出任务级成绩单 → SC 复盘归档 → 用户查看         │
│ 内容：每个 Agent 的表现、谁超前/延期、失败原因、建议     │
└─────────────────────────────────────────────────────────┘
```

#### 1.6.3 用户不可被打扰的原则

1. **执行过程中的技术问题** — Leader 自主处理，不上报
2. **Worker 之间的协调** — Leader 自主处理，不上报
3. **代码缺陷修复（3轮以内）** — Leader 自主处理，不上报
4. **模型切换（同任务类型内）** — Leader 自主处理，不上报
5. **子任务延期（≤2小时）** — Leader 自主评估，不上报
6. **预算微调（≤10%）** — SC 自主审批，不上报用户

**必须打扰用户的场景**（详见 §2.7 权限管理）：
- 计划或方案需要重大变更
- 预算超支 >20%
- 任务卡死 >6小时
- 整个项目面临失败风险

## 📋 第二章：SAS 工作流程

### 2.1 六阶段门控流水线（🔄 v1.4.0 重构）

每个任务必须经历以下六个**逻辑阶段**，但**允许在阶段内并行、在阶段间按规则回退**。

#### 阶段一：计划
**目标**：明确需求，制定计划  
**产出**：项目需求说明书、功能分解、Token/成本/时间估算、风险评估、`maintenance_required` 标志  
**允许并行**：无  
**阶段门**：计划完成后**自动进入方案设计**（无需审批，除非用户标记高风险）  
**状态映射**：`pending`

#### 阶段二：方案设计
**目标**：设计详细方案，确定智能体分配  
**产出**：技术方案、工作流设计、验收标准、智能体分配方案（附分配佐证）  
**允许并行**：设计整体架构后，可**提前启动**部分低风险实施准备（如环境搭建、数据准备）  
**阶段门**：  
- 低风险方案 → **自动进入实施**  
- 高风险（涉及预算超20%、核心架构变更）→ 需用户审批  
**状态映射**：`pending`

#### 阶段三：实施与执行
**目标**：按方案执行，可拆分为多个子任务由不同 Agent 并发处理  
**产出**：代码、配置、中间产物  
**允许并行**：✅ 强烈鼓励，子任务之间无依赖的可同时开工  
**阶段门**：实施完成后**自动触发测试**  
**状态映射**：`running`（可拆分为多个 `running` 子任务）

#### 阶段四：测试与验证
**目标**：确保质量符合要求  
**产出**：测试报告、缺陷列表  
**回退规则**（重要）：  
- 发现缺陷 → 生成**修复方案**（需说明影响范围）→ **可回到阶段三（实施）或阶段二（方案设计）**  
- 最多重试 **3 轮**，超过则暂停并请求人工介入  
**阶段门**：测试通过后进入复盘  
**状态映射**：`running`

#### 阶段五：复盘与归档
**目标**：总结经验，写入案例库  
**产出**：复盘报告、案例卡片（成功/失败）、准则优化建议  
**允许并行**：复盘可与阶段六（若需要）并行进行  
**阶段门**：归档完成后，根据任务类型决定是否进入运维  
**状态映射**：`completed`（归档完成）

#### 阶段六：运维与售后（🔄 v1.4.0 改为可选）
**触发条件**：阶段一的计划中 `maintenance_required: true`  
**目标**：持续监控、异常自动恢复  
**产出**：运维手册、监控配置、异常记录  
**结束条件**：任务生命周期结束（用户主动终止或达到预设期限）  
**状态映射**：`running`（运维中）

### 2.2 自动审批规则（🔄 v1.4.0 新增，v1.5.0 补充复杂度联动）

为减少人工干预，低风险阶段门由**审批 Agent** 自动核准。审批规则与 §2.5 任务复杂度分级标准联动。

#### 2.2.1 审批规则表

| 审批类型 | 自动通过条件 | 复杂度限制 | 否则处理 |
| ---------------------- | ---------------------------------------------- | ---------- | -------------- |
| 阶段门（计划→方案） | 所有任务（除非用户手动标记高风险） | L1/L2/L3 均自动通过 | 否（自动通过） |
| 方案设计（阶段二→三） | Token 估算 ≤ 预算20% + 无架构变更 + 无外部依赖 | L1/L2 自动通过 | L3 需**用户审批** |
| 预算超支申请 | 超支 ≤ 10% 且用户事先授权 | L1/L2/L3 均适用 | SC 审批后上报 |
| 修复方案（阶段四回退） | 影响范围 ≤ 2个子任务 + 修复预估 ≤ 1小时 | L1/L2 自动通过 | L3 需 SC 确认 |
| 运维终止 | 连续3次健康检查失败 | L1/L2/L3 均适用 | 是 |

#### 2.2.2 复杂度联动说明

- **L1（简单）任务**：所有阶段门自动通过，Leader 完全自治
- **L2（中等）任务**：除方案设计涉及预算>20%外，自动通过
- **L3（复杂）任务**：方案→实施阶段门**必须用户审批**，其余自动通过

#### 2.2.3 审批 Agent 行为

自动核准后，在任务日志中记录：
```
approval: auto, reason: {满足的条件}, complexity: {L1/L2/L3}
```

### 2.3 项目任务管理系统对接规范

#### 2.3.3 评分维度与标准

| 维度       | 权重 | 评分标准（1-10）               |
| ---------- | ---- | ------------------------------ |
| **速度**   | 25%  | 响应时间、执行效率             |
| **质量**   | 25%  | 输出正确性、完整性             |
| **问题率** | 25%  | 出错次数、返工次数（越低越好） |
| **成本**   | 25%  | Token消耗、时间成本            |

**综合评分** = 速度分×0.25 + 质量分×0.25 + (10-问题分)×0.25 + 成本分×0.25

#### 2.3.4 四项核心要求

| 要求       | 含义                    | 达标标准          |
| ---------- | ----------------------- | ----------------- |
| **快**     | 响应快、执行快、交付快  | 实际时间≤计划时间 |
| **准**     | 理解准、执行准、交付准  | 一次通过率≥90%    |
| **问题少** | 预判准、风险低、少返工  | 返工次数≤1次/任务 |
| **成本低** | Token省、时间省、资源省 | 实际成本≤预算80%  |

### 2.5 任务复杂度分级标准（🔄 v1.5.0 新增）

**目的**：为自动审批、检索策略、流程走完整版还是快速通道提供量化判断依据。

#### 2.5.1 三级复杂度定义

| 级别 | 名称 | 判断标准（满足任一即为该级别） | 流程要求 | 检索策略 |
|------|------|------------------------------|---------|---------|
| **L1** | 简单 | 单一文件修改 / 单个 API 调用 / 回答问题 / 格式转换 / 不涉及架构 | 快速通道：计划+实施合并，跳过方案设计独立阶段 | 可选检索 |
| **L2** | 中等 | 多文件修改 / 需要测试验证 / 涉及 2-3 个 Agent 协作 / 子任务≤5个 / 预算≤20% | 标准流程：六阶段正常走，阶段内可并行 | 必须检索 |
| **L3** | 复杂 | 架构变更 / 涉及 ≥4 个 Agent / 预算>20% / 跨项目协作 / 外部依赖集成 / 用户标记高风险 | 完整流程：六阶段 + 方案审批必须用户确认 + Leader 全程盯 | 必须检索 |

#### 2.5.2 复杂度判断决策树

```
任务到达
  ↓
涉及架构变更？ ──是──→ L3（复杂）
  ↓否
涉及 ≥4 个 Agent？ ──是──→ L3（复杂）
  ↓否
预算 >20%？ ──是──→ L3（复杂）
  ↓否
涉及多文件/需测试/2-3 Agent？ ──是──→ L2（中等）
  ↓否
L1（简单）→ 快速通道
```

#### 2.5.3 复杂度与自动审批联动

| 复杂度 | 计划→方案 | 方案→实施 | 修复回退 | Leader 权限 |
|--------|----------|----------|---------|------------|
| L1 | 自动通过 | 自动通过 | Leader 自主 | 完全自治 |
| L2 | 自动通过 | 自动通过（预算≤20%） | Leader 自主 | 完全自治 |
| L3 | 自动通过 | **需用户审批** | Leader 自主（≤3轮） | 需上报预算变更 |

#### 2.5.4 级别变更规则

- **升级**：执行中发现复杂度比预估高，Leader 可将 L1→L2（自主），L2→L3（需上报 SC）
- **降级**：执行中发现比预估简单，Leader 可自主降级
- **变更记录**：所有级别变更写入任务日志 `complexity_change: {from} -> {to}, reason: {原因}`

### 2.6 Leader 自治规则（🔄 v1.5.0 新增）

**定义**：Leader Agent 是阶段三（实施）到阶段五（复盘）的实际管理者，在权限范围内拥有完全的自主决策权。

#### 2.6.1 Leader 的产生

| 产生方式 | 适用场景 | 产生者 |
|---------|---------|--------|
| SC 指派 | 大多数任务 | SC 根据任务类型从能力档案中选择 |
| 用户指定 | 高优先级/敏感任务 | 用户在需求中明确 |
| 自动选举 | 简单任务 | 系统根据模型能力自动匹配 |

#### 2.6.2 Leader 可自主决策的范围（L2 权限内）

| 自治事项 | 条件限制 | 记录要求 |
|---------|---------|---------|
| 子任务分解 | 子任务数量不限，但需符合方案设计 | 写入任务清单 |
| Worker 分配 | 参考 §3.4 能力档案，需附分配佐证 | 写入任务日志 |
| Worker 间协调 | Worker 之间有依赖时 | 写入协调记录 |
| 代码修复（自身范围内） | 修复≤3轮 | 写入修复日志 |
| 模型切换（同任务类型） | 参考模型选择策略 | 写入切换日志 |
| 子任务延期评估 | 延期≤2小时 | 写入评估记录 |
| 子任务终止 | 单个子任务失败且不影响整体 | 写入终止记录+原因 |
| Worker 失败重分配 | 原失败 Worker 的任务分给其他 Worker | 写入重分配记录 |

#### 2.6.3 Leader 必须上报 SC 的场景

| 上报场景 | 上报条件 | SC 处理 |
|---------|---------|---------|
| 任务范围变更 | 影响交付目标或时间 | SC 评估是否需要通知用户 |
| Worker 完全失败 | 同一 Worker 重试3轮仍失败 | SC 评估是否换模型/换 Agent |
| 预算超支 | 超20%或超出分配额度 | SC 决定是否申请用户审批 |
| 任务卡死 | 任何子任务>6小时无进展 | SC 介入诊断 |
| 安全/合规问题 | 涉及数据安全、权限越界 | SC 立即上报用户 |
| Leader 无法处理 | Leader 自身能力不足以解决问题 | SC 接管或换 Leader |

#### 2.6.4 Leader 禁止行为

- ❌ **禁止直接联系用户** — 所有用户沟通必须通过 SC
- ❌ **禁止修改计划或方案** — 如需修改，上报 SC 处理
- ❌ **禁止自行决定预算分配** — 超出分配需 SC 审批
- ❌ **禁止删除其他 Worker 产物** — 即使认为错误，也需标记而非删除
- ❌ **禁止修改 SAS 准则** — 准则变更需走迭代流程

### 2.7 Agent 权限管理（🔄 v1.5.0 新增）

**目的**：为所有参与任务的 Agent 建立统一的权限体系，防止越权操作，确保责任清晰。

#### 2.7.1 四级权限模型

| 权限等级 | 持有者 | 定义 | 说明 |
|---------|--------|------|------|
| **L0** | 用户（User） | 最高权限，所有决策的最终审批者 | 包括但不限于：审批计划、审批方案、终止项目、修改准则 |
| **L1** | SC（主脑） | 任务级管理权限 | 制定计划、设计方案、指派 Leader、质量终审、预算微调（≤10%） |
| **L2** | Leader Agent | 子任务级管理权限 | 任务分解、Worker 分配、进度监控、问题处理（L3 内）、模型切换 |
| **L3** | Worker Agent | 执行级权限 | 仅限分配给自己的任务范围，不跨界 |

#### 2.7.2 权限矩阵

| 权限项 | 用户(L0) | SC(L1) | Leader(L2) | Worker(L3) |
|--------|---------|--------|------------|-------------|
| **审批计划** | ✅ | ❌ | ❌ | ❌ |
| **审批方案** | ✅ | ❌ | ❌ | ❌ |
| **制定计划** | ❌ | ✅ | ❌ | ❌ |
| **设计方案** | ❌ | ✅ | 📋辅助 | ❌ |
| **指派 Leader** | 📋建议 | ✅ | ❌ | ❌ |
| **分解子任务** | ❌ | ❌ | ✅ | ❌ |
| **分配 Worker** | ❌ | ❌ | ✅ | ❌ |
| **Worker 执行任务** | ❌ | ❌ | ❌ | ✅ |
| **代码修复（自身范围）** | ❌ | ❌ | ❌ | ✅ |
| **代码修复（跨任务）** | ❌ | ❌ | ✅ | ❌ |
| **Worker 间协调** | ❌ | ❌ | ✅ | ❌ |
| **模型切换（同任务类型）** | ❌ | ❌ | ✅ | ❌ |
| **任务延期评估（≤2h）** | ❌ | ❌ | ✅ | ❌ |
| **预算超支申请（≤10%）** | ❌ | ✅ | 📋建议 | ❌ |
| **预算超支（>20%）** | ✅ | ❌ | ❌ | ❌ |
| **修改计划/方案** | ✅ | ✅ | ❌ | ❌ |
| **终止子任务** | ❌ | ✅ | ✅ | ❌ |
| **终止整个项目** | ✅ | ❌ | ❌ | ❌ |
| **联系用户** | ✅ | ✅ | ❌ | ❌ |
| **读取案例库** | ❌ | ✅ | ✅ | 📋按需 |
| **写入案例库** | ❌ | ✅ | 📋建议 | ❌ |
| **修改 SAS 准则** | ✅ | ❌ | ❌ | ❌ |
| **删除其他 Agent 产物** | ✅ | ✅ | ❌ | ❌ |

> ✅ = 完全权限  📋 = 建议/辅助权  ❌ = 无权限

#### 2.7.3 越权处理机制

| 越权等级 | 定义 | 检测者 | 处理方式 |
|---------|------|--------|---------|
| **低风险** | Worker 读了非分配范围的文件、Leader 微调了不涉及核心的参数 | Leader（Worker越权）/ SC（Leader越权） | 记录越权事件到成绩单 + 警告该 Agent + 任务继续 |
| **中风险** | Leader 未上报就扩大了任务范围、SC 未通知用户就修改了方案 | SC | 记录越权事件 + SC 评估影响 + 必要时回滚 |
| **高风险** | 任何 Agent 尝试联系用户、修改准则、删除关键数据 | 系统/SC | 立即停止该 Agent 所有操作 + SC 上报用户 + 记录到案例库作为反面教材 |

**越权记录模板**：

```yaml
越权事件:
  时间: "{YYYY-MM-DD HH:MM}"
  任务ID: "{任务唯一标识}"
  越权Agent: "{Agent名称}"
  权限等级: "{L1/L2/L3}"
  越权行为: "{具体做了什么}"
  风险等级: "{低/中/高}"
  处理方式: "{警告/回滚/停止+上报}"
  影响评估: "{对任务的影响}"
```

#### 2.7.4 权限检查执行点

| 执行时机 | 检查内容 | 检查者 |
|---------|---------|--------|
| Leader 产生时 | 确认 L2 权限范围已告知 | SC |
| Worker 分配时 | 确认 Worker 只收到自己任务的上下文 | Leader |
| 每次阶段切换时 | 确认阶段切换权限（见 §2.1 阶段门） | Leader |
| 模型切换时 | 确认在 L2 权限内 | Leader |
| 预算变更时 | 确认在 L1 权限内（≤10%）或 L0（>20%） | SC |
| 任务完成时 | 检查全流程是否有越权记录 | SC + Leader |

### 2.9 任务紧急度分级处理规则（🔄 v1.6.0 新增）

**目的**：区分任务优先级，确保高优先级任务不被低优先级任务阻塞。

#### 2.9.1 紧急度定义

| 紧急度 | 定义 | 触发条件 | 响应时限 |
|--------|------|---------|---------|
| **P0-紧急且重要** | 必须立即处理 | 用户明确标记"紧急"；预算超支>50%；任务链断裂风险>80% | 1小时内响应 |
| **P1-重要** | 尽快处理 | 涉及用户关键决策；预算消耗>80%但任务未完成；多个子任务延期>2h | 4小时内响应 |
| **P2-普通** | 正常流程 | 无特殊风险，正常任务 | 按计划处理 |
| **P3-低优先级** | 可延后 | 优化类任务；文档整理类 | 可批量处理 |

#### 2.9.2 处理规则

| 规则 | 说明 |
|------|------|
| **P0打断P2/P3** | 立即暂停P2/P3任务，切换到P0 |
| **P1优先于P2/P3** | P1在队列中排在P2/P3前面 |
| **P2可并行** | 多个P2可同时执行 |
| **P3可批量** | 同类P3任务合并执行 |

#### 2.9.3 紧急度标记方式

- 任务名称前缀：`[P0]`/`[P1]`/`[P2]`/`[P3]`
- 飞书消息中注明紧急度
- 计划文档中标注

#### 2.9.4 紧急度升级规则

| 升级条件 | 升级结果 | 通知 |
|---------|---------|------|
| P2任务延期>4小时 | → P1 | 自动发送飞书通知 |
| P1任务延期>2小时 | → P0 | 自动发送飞书通知 |

## 🚨 第四章：异常监控与恢复 (v1.3.0 升级，v1.4.0 补充)

### 4.1 任务生命周期管理规范

**目的**：解决任务链断裂（pending状态无人接管）问题

#### 4.1.1 任务状态机

pending → running → completed
↓
failed → 可重试?
↓
cancelled

| 状态          | 定义                       | 超时限制             |
| ------------- | -------------------------- | -------------------- |
| **pending**   | 任务已创建，等待智能体接管 | **≤2小时**，超时告警 |
| **running**   | 智能体正在执行             | ≤计划时间×1.5        |
| **completed** | 任务成功完成               | —                    |
| **failed**    | 任务执行失败               | —                    |
| **cancelled** | 任务被取消                 | —                    |

#### 4.1.2 任务看门狗机制

| 检查项              | 阈值      | 动作                   |
| ------------------- | --------- | ---------------------- |
| **pending超时告警** | >2小时    | 发送告警到飞书群       |
| **pending自动升级** | >6小时    | 触发自动重试或升级处理 |
| **running超时预警** | >计划×1.2 | 发送预警通知           |
| **running超时中断** | >计划×1.5 | 暂停任务，等待人工决策 |

#### 4.1.3 阶段切换日志要求（v1.4.0 补充）

每个阶段切换时，必须在任务日志中记录：

phase: {from} -> {to}
gate_decision: auto|manual
reason: {简短原因}

示例：

phase: design -> implementation
gate_decision: auto
reason: 低风险方案，自动审批通过

#### 4.1.4 自动恢复机制

**原则**：会话重置后，未完成任务必须在下次启动时自动恢复

任务恢复流程：

1. 检测重置事件（通过 commands.log 或 heartbeat 异常）
2. 查询任务队列状态（/api/harness/tasks）
3. 识别 pending/failed 状态任务
4. 评估恢复可行性（超时时间、错误原因）
5. 执行恢复动作：
   - pending>6h → 重新创建任务
   - pending<6h → 等待原智能体恢复
   - failed → 分析错误原因后重试
6. 记录恢复日志

### 4.2 智能体空转检测规范

**目的**：解决智能体空转（大量心跳快照但0 Token工作产出）问题

#### 4.2.1 空转定义

**空转状态**：连续3小时快照数>100但Token输入=0

#### 4.2.2 空转检测指标

| 指标           | 正常范围        | 告警阈值      |
| -------------- | --------------- | ------------- |
| 快照频率       | 10-50个/小时    | >100个/小时   |
| Token输入/快照 | >100 Token/快照 | =0 Token/快照 |
| 连续空转时长   | 0               | >3小时        |

#### 4.2.3 空转处置流程

检测到空转 →
↓

1. 记录空转起始时间和快照数
2. 发送告警到飞书群（包含持续时长和快照数）
3. 检查任务队列状态
4. 检查 API 连通性
5. 如持续>6小时，尝试触发任务重新分配
6. 记录诊断信息和处置动作

### 4.3 会话重置恢复SOP

**背景**：86%的会话重置由 `gateway:sessions.reset` 自动触发，需标准恢复流程

#### 4.3.1 五步恢复流程

| 步骤        | 操作                                                 | 目标状态 |
| ----------- | ---------------------------------------------------- | -------- |
| **1. 检测** | 监控 commands.log 中的 `gateway:sessions.reset` 事件 | 识别重置 |
| **2. 快照** | 保存当前任务队列快照到内存/文件                      | 防止丢失 |
| **3. 等待** | 等待30秒确认重置完成                                 | 稳定状态 |
| **4. 恢复** | 从 memory/sessions 加载上下文，继续未完成任务        | 继续执行 |
| **5. 验证** | 确认任务链完整性和Token产出                          | 恢复正常 |

#### 4.3.2 重置原因分类

| 原因                   | 占比 | 处理策略             |
| ---------------------- | ---- | -------------------- |
| gateway:sessions.reset | ~86% | 自动恢复，检查任务链 |
| 飞书渠道重置           | ~5%  | 检查消息丢失         |
| 人工重置               | ~5%  | 确认任务状态         |
| 其他                   | ~4%  | 按情况分析           |

### 4.4 任务失败处理流程



任务执行失败 → 记录失败信息+错误日志 → 评估失败原因
↓
├── 可重试（网络超时、偶发错误）
│ → 等待5分钟后重试（最多3次）
│ → 重试成功 → 继续执行
│ → 重试失败 → 进入不可重试流程
│
├── 需人工介入（逻辑错误、配置问题）🐾主脑负责
│ → 立即暂停任务
│ → 记录详细错误信息
│ → 通知用户说明问题
│
└── 需修改方案（需求不清、技术不可行）🐾主脑负责
→ 暂停任务
→ 提出替代方案
→ 等待用户确认

### 4.5 预算超支处理

| 预算消耗 | 等级   | 处理方式                         |
| -------- | ------ | -------------------------------- |
| <80%     | 🟢 正常 | 正常执行，注意监控               |
| 80-90%   | 🟡 预警 | 发出预警通知，使用低成本模型     |
| 90-95%   | 🟠 警戒 | 暂停非关键任务，重要任务需确认   |
| >95%     | 🔴 紧急 | 只执行紧急任务，立即通知用户     |
| 超支     | ⚠️ 超出 | 停止所有非授权任务，等待用户指示 |

### 4.6 响应超时处理

| 超时情况         | 处理方式                 |
| ---------------- | ------------------------ |
| API响应超时2分钟 | 自动重试（最多2次）      |
| 重试仍超时       | 标记"卡住"状态，通知用户 |
| 30分钟无法恢复   | 提供替代方案或降级方案   |

### 4.8 任务风险预测机制（🔄 v1.6.0 新增）

**目的**：从事后告警升级为事前预测，提前发现风险，有时间准备预案。

| 预测类型 | 触发条件 | 预测因子 | 预测输出 |

#### 5.3.2 成绩单产出流程

### 5.4 SAS体系效能衡量指标（🔄 v1.6.0 新增）

**目的**：系统化衡量 SAS 体系运作效果，量化价值投入产出比，形成数据驱动的迭代闭环。

#### 5.4.1 定量指标体系

| 指标名称 | 计算方式 | 目标值 | 告警阈值 |
|---------|---------|-------|---------|
| **任务成功率** | 成功任务数/总任务数 | ≥85% | <70% |
| **任务准时率** | 准时完成任务数/总任务数 | ≥80% | <65% |
| **预算控制率** | 实际成本/预算成本 | ≤90% | >100% |
| **平均任务周期** | 总实际时间/任务数 | ≤计划时间×1.2 | >计划时间×1.5 |
| **返工率** | 返工次数/总任务数 | ≤15% | >25% |
| **案例库填充率** | 新增案例数/完成任务数 | ≥50% | <30% |

#### 5.4.2 定时报告机制

| 报告类型 | 周期 | 包含内容 |
|---------|------|---------|
| **SAS周报** | 每周五 | 上述6项指标 + 环比变化 + TOP3问题 |
| **SAS月报** | 每月末 | 趋势分析 + 改进建议 + 下月重点 |
| **SAS季报** | 每季度 | 全面复盘 + 用户决策参考 |

#### 5.4.3 报告格式模板

```yaml
SAS效能报告:
  报告类型: "{周报/月报/季报}"
  周期: "{起止日期}"
  
  指标汇总:
    - 指标名称 | 目标值 | 实际值 | 状态(🟢达标/🟡接近/🔴未达标)
  
  环比变化: vs上周/上月
  主要问题: TOP3
  改进建议: 对应准则优化方向
```

#### 5.4.4 指标异常处理

| 状态 | 条件 | 处理方式 |
|------|------|---------|
| 🟢 达标 | 实际值达到或超过目标 | 保持现状，记录成功经验 |
| 🟡 接近 | 实际值在目标±10%以内 | 增加监控频率，分析原因 |
| 🔴 未达标 | 实际值偏离目标>10% | 立即分析原因，纳入下一版本优化 |

### 6.3 Token产出监控规范

